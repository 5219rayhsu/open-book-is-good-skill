# 詳解生成 × 紅隊查核 × 作文政策

> 本檔是「開卷有益・升學」復刻手冊的一節，給載入 skill 的 agent 與想復刻／貢獻的人。
> 主題：怎麼用 clean-room 方式自行生成題目詳解、怎麼用必跑的反駁式紅隊（全跑或抽查）查核品質、
> 以及為什麼作文不用 AI 代寫。授權 MIT。

---

## 0. 一句話總綱

**自己生成詳解、絕不抄出版商；每則標「AI 整理需查證」＋ 把握度。詳解生成（選擇題＋申論題）後，紅隊查核「必跑、不可跳過」，執行前問使用者要「全跑」還是「抽查」；抽查時用統計停止規則決定要不要升級更強模型；只修確認的錯。作文不 AI 代寫，改抓官方分級樣卷。**

> 另有一道與本檔並行的必跑門在解析端：**答案配對全驗** —— 每題答案逐題對官方標準答案核對、全跑不抽樣（見 `SKILL.md` 必跑門 A）。本檔談的「抽查」只適用詳解紅隊，**不適用答案配對**。
>
> 還有一道由**題文修正**觸發、不是由生成流程本身觸發的硬門：**修題必檢詳解**——`bank.json` 任一題的題幹／選項／答案被修正時，該 qid 既有詳解一律送檢（答案錨重跑＋內文與新題文逐項對照，不符即整則重寫），無詳解則記入待寫清單；修復報告須逐筆標記詳解影響。規則本體與案例見 `dirty-data-robustness.md` 第四節「修題必檢詳解（衍生物一致性）」。

## 1. clean-room 詳解：自行生成，不餵出版商詳解

### 1.1 為什麼是 clean-room

依著作權法 §9.1.5，「依法令舉行之各類考試試題」不受著作權保護，可自由重製 —— 所以題幹、選項、官方標準答案都能收。**但出版商的編排與詳解受著作權保護**，不能抄、也不能餵進生成 prompt 當參考。

因此詳解一律 **clean-room 自行生成**：

- 輸入只有「題幹 ＋ 選項 ＋ 官方標準答案鍵」這些公共所有的素材。
- **不把出版商的詳解寫法、解題步驟、口訣餵進 prompt**，避免衍生出受保護編排的近似品。
- 官方標準答案 PDF 是**權威答案鍵**（用來校準正解），但「為什麼選這個」的論述由模型自己長出來。

### 1.2 每則詳解的硬性標記

> ⚠️ 鐵則：**任何詳解都不是官方標準答案**。AI 會錯，使用者必須能一眼看出這是「AI 整理」而非權威定論。

正本 `data/explanations.json` 的 `meta.note` 固定寫明這件事（實際字串）：

```
AI 整理的本題解釋，非官方標準答案，須對照現行課綱與權威資料查證。
```

`meta.source` 為 `開卷有益升學：學測+會考`，`meta.count` 為詳解總數（merge 時自動同步）。

每一則詳解 entry 的結構（**只有兩個欄位，刻意極簡**）：

```json
{
  "t": "詳解全文（繁中、全形標點）",
  "c": "high | med | low"
}
```

- `t`（text）：詳解本體。
- `c`（confidence，把握度）：模型對這則詳解的把握度，三級：
  - `high` —— 有把握（多為單純字音字義、明確法條、定義題）。
  - `med` —— 中等（需要推論、跨段整合）。
  - `low` —— 把握度低（争議題、需查現行課綱、模型自覺不確定）。
- 前端依 `c` 給使用者「把握度」提示；把握度低的更要使用者自行查證。

> 實際分佈（2026-06 升學題庫 1443 則）：`high` 1119、`med` 275、`low` 49。把握度不是裝飾，是誠實揭露 —— 多數題模型有把握，但確實有一小撮自承不確定，這些必須讓使用者看見。

### 1.3 鍵（qid）與 merge

詳解以 **qid 為鍵**存進 `explanations.json` 的 `explanations` 物件：

- 學測：`115_國綜_1`、`114_社會_23`（`{年}_{科}_{題號}`）。
- 會考：`會考_113_社會_19`（`會考_{年}_{科}_{題號}`）。

merge 一律 **qid-keyed、冪等**：同一 qid 重跑就覆寫，整個 entry 取代舊值，只保留 `{t, c}`、丟棄任何殘欄（例如空殼遺留的 `text`）。寫入正本的腳本是唯一入口（見 `scripts/merge_stage3.py`），不讓多個程序同時寫 `explanations.json`。

### 1.4 三段固定結構（單選、複選一體適用）

只靠「先點正解、再逐一辨析選項」這種通則寫多了會走鐘——辨析段落黏成一坨、考點漏講、前端渲染不出段落感。實測收斂出的硬規則是：每則 `t` 固定分三段，**段間以空行（連續兩個換行）分隔**，對應前端 `.explain-body` 的 `white-space:pre-line`（`\n\n` 才會呈現為真正的空行，藉此把三段在視覺上清楚隔開）：

1. **正解段（單獨自成一段）**：先明講哪個／哪些選項正確、為什麼。複選題逐一講每個正確選項的理由；計算題的逐步演算放在這段。段首用「正解：」樸素標記，不用 markdown 符號。
2. **選項辨析段（每個選項各自成段）**：逐一講其餘選項為何錯，深度依「誘答是否為實質概念」分流（見下）。**每個選項的辨析獨立成一段、段與段之間空行分隔、每段以該選項字母起頭**（如「A：」「B：」，選項一律用大寫字母 A–E，禁用原卷的數字編號——考生看畫面標記是字母，詳解卻用數字指稱會對不上），不再重複冠上「選項辨析：」字樣。
3. **本題考點段（末段、前面空一行）**：以「本題考點：」開頭，列 1–3 個概念／理論／專有名詞（有通用英文原名的附上），一句話點出它是什麼。

**字數**：三段完整優先於字數通則；**概念辨析型以 600 字為度，超過即檢討是否贅述**；**計算題不受此字數限制**，以「步驟完整、考生可重現」為準（每個誘答選項都要算出它錯在哪一步、哪個數字，不得以「基礎不同故不符」一語帶過），但不得夾帶與解題無關的贅述。正文須為乾淨成品，**不得夾帶修正註記、審查對話、自我驗證過程**（如「（更正…）」「經查…」這類鷹架）；法規題可引用現行法條號或官方公開命題／評分說明當查證錨點——這屬公共領域、非受保護編排，clean-room 只禁抄出版商的解題編排與口訣，不禁引一次來源。

**申論題不適用本節分流**：申論無標準答案鍵，走不了答案配對硬門，一律由**最強模型**直接撰寫參考要點／評分向度（見 §3.4），不進本節這套「先生成、後紅隊」的分工。

### 1.5 合併解析：同因才合併，不是為了省字

多個選項（不論正解或誤答）若判斷理由本質相同、分開寫必然逐句重複，才合併為同一段——**目的不是省字，是分開寫反而讓考生看不出差異**。

- **合併門檻嚴格限縮**：僅限「同一判準／同一條法規／同一個概念錯位」導致的同因情形。理由不同、僅結論同為錯的選項一律各自分段。
- **計算題各誘答錯在不同步驟或數字者屬本質不同，不得合併**——這是計算題的硬性排除，每個誘答都要能各自算出它錯在哪。
- **合併後仍須逐一點出字面差異**：段首以該組選項字母依選項順序、以頓號「、」相連再加全形冒號起頭（如「A、C：」，與複選題正解段列舉多個正確選項時的字母寫法一致），段內不得只寫一句籠統結論帶過——即使錯因相同，也要讓考生分得出「A 錯在哪、C 又錯在哪」。

**誘答辨析深度依題型分流**：純數字／單一事實記憶題（誘答僅為替代數字或年份）講清正解依據即可、可低於 120 字；但當誘答本身是實質概念（易與正解混淆的學說、方法、分類）時，必須逐一點出正解與各誘答的關鍵差異，並說明「為什麼這個誤答看起來對」，不得只寫「屬 X，故選」。

## 2. 模型路由：機械便宜、文字中階、難題最強

詳解生成不是「全程開最貴模型」。依成本守則分三檔，把錢花在真正需要推理的地方：

| 工作 | 模型檔位 | 例子 |
| --- | --- | --- |
| 機械工作 | 小模型（Haiku） | 下載 PDF、轉檔（markitdown）、檔案搬移 |
| 文字科解析／詳解 | 中階（Sonnet） | 國文／英文／社會的逐題詳解、選項辨析 |
| 題組長文／難題／圖式／紅隊 | 最強（Opus） | 閱讀題組長文理解、爭議難題、看圖判讀、反駁式紅隊查核 |

理由：成本與品質平衡，並**避免「便宜生成 → 昂貴查核」反而總成本更高的成本倒置**。查核（紅隊）固定用最強模型，**且必跑**；至於**全跑（逐題）還是抽查（統計停止規則）由使用者拍板**（見下節）。

> 升學分支的實測校準（國文／英文／社會）：Sonnet 生成 + Opus 抽驗，累計抽樣的真錯率都遠低於升級門檻，故各科**續用 Sonnet** 生成。這正是統計停止規則要回答的問題 —— 什麼時候該換更貴的模型。

### 2.1 品質拉齊：讓執行模型的產出趨近強模型水準的五個機制

紅隊（§3）抓的是「對錯」，這裡要拉齊的是**教學品質**——量產詳解的模型（下稱**執行模型**）品質難免比直接下筆的**強模型**鬆散。與其事後靠紅隊逐則挑出鬆散，不如把拉齊機制內建進生成流程本身。以下五個機制是固定組合，文字表述可依實作微調，但機制本身不建議刪減：

1. **金樣 few-shot**：由強模型依 §1.4／§1.5 全部條款先寫 3–5 則「金樣詳解」（涵蓋不同題型：概念辨析、計算、複選），生成時作為 few-shot 範本塞進執行模型的 prompt；配題**優先取同科同題型**的金樣，讓執行模型模仿與待生成題最相近的示範。
2. **結構 lint（零 token 硬門）**：生成後先跑純腳本檢查（**不動用任何模型**）：首段「正解：」與末段「本題考點：」標記齊全且順序正確、中段每段以一個或多個 A–E 字母（多個時以頓號相連，即合併解析段）加全形冒號起頭、段落以空行分隔、每個非正解選項字母在中段恰出現一次（不論獨立段或合併段）、選項指稱只用 A–E、無鷹架詞（更正／經查等審查對話殘留）、計算題至少含一步「數字＝數字」演算、字數下限。**任一項不過就整則退回重生，不送紅隊**——結構有問題的則送紅隊只是浪費一次查核額度。
3. **檢索錨定**：prompt 隨題附上該科現行法條清單／公式表／課綱藍圖的相關片段，讓執行模型的引用有現成依據可對照，而非僅憑訓練記憶覆誦，降低杜撰法條號或引用舊法規的機率。
4. **分級路由**：預判為把握度偏低、法規時效敏感、或官方曾發布釋疑／送分的題，**一開始就交強模型撰寫**，不進執行模型的量產流程；申論題本就一律強模型撰寫（見 §1.4 末），不受本條額外影響。
5. **審計回饋圈**：每批完成後，強模型抽一定比例做終審，把發現的錯誤模式歸納成「常見雷」清單，回灌下一批執行模型的 prompt，讓品質逐批收斂，而不是每批從頭重踩同樣的坑。

### 2.2 科目特定注意（撰寫詳解前先查）

個別科目有各自的命題陷阱，撰寫與紅隊時先過這張清單；每條一行，持續累積。

- 地理（學測社會等）：地圖題的指北箭頭**可能不朝上**（如 111 學測社會圖 6 指北朝左）——方位描述必須先讀圖上指北，別假設上北下南；這常是命題陷阱本體。

## 2.3 pipeline token 工程：大批量時把「省 token」當品質槓桿

當題庫大到「數萬則詳解、數百批、跨越額度重置的多日背景工程」，prompt 的**固定重複稅**（每個生成 agent 都重載一次規範）與**紅隊輸出量**會主宰總成本。此時有一組槓桿能把「省 token」與「升品質」對齊——不是取捨。先立**三層分工**（貫穿本節）：**結構 lint 抓格式、檢索錨定防事實錯（讓模型抄現行條文而非憑記憶）、紅隊抓剩下的事實錯**。token 工程省的是「前兩層零 token 手段能接手的部分」，**絕不是弱化紅隊**——紅隊必跑、必抽，抽樣率只增不減。兩個總原則：（一）省 token＝升品質是雙贏；（二）**抽樣率是品質參數、輸出格式是成本參數，分開調**。

> 給復刻者：以下限定的數據來自一次真實戰役——某國考社工師科 6,919 題「舊詳解結構重構」，抽審約 1,650 則、揪出 166 則真錯（≈10% 真錯率），據此校準了下面幾條限定。你的科目分佈不同，數字會不同，但「錯誤集中在新寫的事實層」這個結構性教訓通用。

**共通七槓桿**

1. **恆定前綴吃 prompt cache**：把每個生成 agent 的 prompt 拆成「恆定前綴＋變動尾」。前綴＝操作卡（槓桿 2）＋1 則**已過紅隊**的同科同題型金樣＋**科級錨定素材包**（把該科法條清單／公式表／課綱片段做成整科共用的常數包，**法規科放現行條文全文**。**素材包有上限**：按紅隊錯誤統計選常引法規優先收錄，總量以 10–20k token 為度——全文包無上限會反噬 cache 經濟，未命中時每批付全額；包外引用一律標低把握度觸發 100% 紅隊，不得憑記憶硬寫）；前綴須逐位元組相同才吃得到 prompt cache，批次題目只放變動尾。配套紀律：**同科連續跑、不跨科交錯**——一跨科前綴就變、cache 失效。
2. **操作卡取代整份規範進 prompt**：生成 agent 不吃規範全文，只吃一張 300–500 token 的**蒸餾操作卡**（三段結構模板＋硬規則，見 §1.4）＋1 則金樣。金樣塞 3–5 則反而稀釋遵循率，取一則最相近即可；規範全文只留給紅隊裁決者與 lint 開發。這是「省 token＝升品質」的雙贏典型。**配套：操作卡×lint 覆蓋矩陣**——規範的每一硬條款必須被「操作卡」或「lint 檢查項」至少其一覆蓋（表列對照），防蒸餾時漏掉 lint 驗不到的內容規則（如合併解析的同因門檻、字數為度）。
3. **lint 由主迴圈直接跑，不另開 lint agent**：生成 workflow 只負責「生成→落 draft 檔」，回到主迴圈後用一條 shell 對整波 draft 跑**結構 lint 腳本**，主迴圈只看 pass/fail 計數。數百批規模下，替 lint 另開一個 agent 的殼開銷（載入指令的固定稅）可達數十萬 token，純浪費。
4. **紅隊 schema 化＋批次化**：fresh-context 的意義是**與撰寫者隔離**，不是「一題一 agent」。同一紅隊 call 批 10–20 題（題目彼此獨立、無交叉污染），用 structured output 強制輸出並**依判決分層**（一刀切「reason 限 30 字」會餓死裁決者與回饋圈兩個下游）：`verdict=ok` **零 reason**（九成題在此省最多）；`verdict=fix` 給 `error_type` 枚舉（法條號｜法定數字｜虛構規定｜權責用語｜免責收尾｜其他）＋ `detail≤100字`——枚舉讓錯誤模式統計直接走腳本 tally，detail 足供裁決者判真錯／誤報。反駁式立場照樣寫進 prompt。這是壓輸出格式（成本參數），不動抽樣率（品質參數）。

**重構線專屬（把舊詳解重排成新結構時，非從零重寫）**

5. **內容守恆檢查（零 token）**：抽出舊詳解的**關鍵 token 集**（數字、法條號／函釋字號、專有名詞、各選項字母的結論詞），驗證重構後的新詳解**全數保留**——防模型重排時默默改寫已驗證的推理（事實漂移）。**管線順序（防自相殘殺）**：腳本預清理（剝免責句、選項編號轉字母）**先跑，守恆基線建在預清理後的舊詳解上**——否則合法刪除全被誤判成漂移；關鍵 token 集**排除選項編號數字**，並做數字正規化（全形↔半形、中文數字↔阿拉伯數字視為同一 token），否則誤殺率會爆表。**每道自動退回門設重試上限（如 2 次）**，仍不過即轉人工佇列，防誤殺死循環。**下探限定（重要）**：守恆檢查原理上**只保舊內容不漂移，抓不到「新寫辨析」裡的事實錯**；社工師實測，真錯正集中在補寫的法律事實層（法條號張冠李戴、法定數字錯如 7 日→10 日、虛構規定、權責用語錯如偵查／調查）。故紅隊抽樣**下探僅適用「淨新增內容占比低」的題**（舊詳解已含完整辨析、重構近乎純重排）；**法規科、以及原本就過短、要補很多新辨析的科，一律不下探、維持較高抽樣（如 30–40%）**。**預設全抽原則（後續裁定，位階更高）**：若你的價值排序是「正確性對使用者＞成本」（學習站多半應如此），把**預設設為 100% 全抽**、部分抽樣當「實測掙來的例外」——降抽條件為零 token 層全部生效＋連續批實測真錯率壓到門檻以下（如 ≤2–3%），任一批回升即回 100%。教訓：實測 10% 真錯率下只抽 24%，等於未抽部分可能藏著數百則錯直接上線——**抽樣的前提是首過品質已被實測證明夠好**，順序不能反。分層 schema（ok 零 reason）已把全抽代價壓掉大半，別拿舊成本想像嚇自己。
6. **格式修不重審**：紅隊已過、只有結構 lint 掛掉的則（純格式），修完只回 lint＋守恆檢查，**不再進紅隊**——內容沒動、守恆檢查即可證。**格式修定義收窄（重要補洞）**：格式修＝**不增刪內容 token 的修復**（段落重排、空行、冒號、字母標記）；lint 掛「字數下限」「缺選項辨析」屬**內容缺陷**，回內容流重構＋照常紅隊，不得當格式修處理。下放便宜模型時兜底網須為**雙向守恆**：舊 token 全保留（防漏）＋**新增內容 token 比例護欄**（防模型為過門自行補內容——單向守恆驗不到「多寫」，那是會讓未審事實直通入庫的洞）。

**新寫線專屬（補全缺口，一開始就用新結構寫）**

7. **指紋聚類、一份共用**：新寫池啟動前，先用**內容指紋（題幹＋選項＋答案；絕不 stem-only**——同題幹異選項是真不同題，見 `shared-question-banks.md`；**指紋必須納入題組前文 passage**——同題幹＋選項＋答案但一在題組內、一為獨立題者，詳解會引用情境脈絡、不可共用，最保守作法是題組成員題直接排除共用）做零 token 聚類，量**同一科跨年度**完全相同題的重複率；每簇只寫一份、整簇套用。跨科重疊常趨近 0，重複多發生在同科跨年；聚類腳本近乎免費，**先量再決定**投入。

**兩條配套硬規則（社工師實測補的教訓）**

- **金樣本身必須先過紅隊才可入 few-shot**：範本自己若埋錯會**系統性汙染同主題題目**——實測有一則金樣把某法條號寫錯，連累了同主題的數則生成。未查核的金樣是放大器，不是護欄。
- **法規科的錨定素材包要放現行條文全文**（不只法條清單）：讓模型「抄」現行條文，而不是「憑記憶引用」。實測五類事實錯裡有三類，源頭都是憑記憶寫法律事實（法條號、法定數字、權責用語）。這正是三層分工中「錨定防事實錯」那層的具體手段。
- **法條引用核對器（零 token，法規科；素材包的免費副產品）**：素材包既已備現行條文全文，腳本即可抽詳解中的「法名＋條號（＋關鍵數字）」引用去比對——條號不存在（虛構條號）直接攔下；條號存在但引用的關鍵數字（日數、金額、年限）不在該條文文字中則 flag。實測五類事實錯有三類可被它預攔或標記。**配套改風險加權抽樣**：被 flag 的題 100% 進紅隊、乾淨題走基準抽樣——同樣紅隊預算，召回率更高。**v2 候選建議**：核對器不只判錯，還在該法全文裡檢索詳解引用的關鍵詞、給出候選正確條號——裁決者從「自己查」變成「確認腳本給的候選」，裁決輸出縮成是／否。
- **素材包本身的建置與時效也腳本化**：從官方法規資料庫抓現行條文全文＋版本戳記（`--build`）；`--check` 比對各法最新異動日期、過期即擋——設為**每波生成的啟動閘**（過期先更新再生成，修法不會漏超過一波）；另設**月維護例行**：每月拿新包對**已上線**詳解跑核對器全量掃、輸出修法影響清單——已發佈的內容也會因修法過時，這是很多題庫站漏掉的維護面。
- **裁決後修復三層（directive 結構化後，正確值已知＝驗證網存在）**：裁決出結構化 directive `{wrong→correct, scope}`——`scope=token`（條號／數字純替換）走**腳本 swap 零 token 修完**、複驗「含正確值、不含錯值」；`scope=sentence` 交便宜模型改寫單句（expected-token 檢查＋雙向守恆兜底）；整段以上大改才交中階模型。**發包分組**：修復一律新代理（不回原生成代理——會帶著原錯誤思路）、**同科×同錯誤類型分組、10–30 題/批**——同科共用素材包前綴吃 cache、同錯類共用修復指引。
- **紅隊側同樣做 token 工程**：（一）紅隊規範＋反駁立場文字做成 **byte-identical 恆定前綴**、同科連跑（生成側的 cache 紀律同樣適用於審者）；（二）**法規科紅隊也附同一份素材包**——審者查法條也不得憑記憶，否則自己記錯就變誤報或漏報；（三）**核對器 flag 題與乾淨題分開批**：flag 批附機器疑點定向查證、乾淨批不帶任何提示——防錨定效應；（四）**短路順序**：先驗「結論是否支持官方答案」，講反即出 fix 不必查其餘；（五）查點補一項**整題張冠李戴**——整則詳解講的根本不是這一題（結構與事實各自成立、卻對不上題幹），lint 與答案錨都驗不到；（六）lint 預掃**結論一致性 regex**（「故選Ｘ」與 `ans` 比對），最嚴重的「講反」先讓機器攔一部分。

**別忘了把這條加進結構 lint**：實測**最大宗**的錯誤類別，是把**免責話術或審查旁白寫進正文**（「以現行條文為準」「以官方答案為準」「信心設為低」之類，數十則）。這類純 lint 就能零 token 攔下——把它們做成**黑名單樣式**加進結構 lint（§1.4 已禁「鷹架詞」，這是同一類的擴充），好讓紅隊火力集中在只有它能抓的事實錯。**修復也零 token，但剝句有安全條件**：只剝「段尾＋短句＋句內無選項字母／數字／法條號」的黑名單句——防黑名單片語出現在實質內容句裡（如題目本身就在考「依現行條文」）被整句誤刪；腳本依句界剝除後重跑 lint＋守恆即證只刪該句，不符安全條件者才交模型修。

**量測（併入試點 golden-batch）**：除了每則 token、lint 首過率、紅隊真錯率，另量四項——prompt cache 命中率（驗前綴設計）、守恆檢查誤殺率（驗關鍵 token 集抽取沒過嚴）、「便宜模型 vs 中階模型」在重構線的**分子集** A/B（拆「純重排子集」與「補寫子集」分別測——混測會讓補寫子集失分淹沒純重排勝場、得出「便宜模型不能用」的假結論；子集分類由腳本啟發式判定，**疑者一律歸補寫子集**；便宜模型只在實測勝出才下放，補寫辨析是真內容生成、非機械重排），以及**摻毒批紅隊召回率**——在試點批植入已知錯誤（法條號改錯、法定數字改錯、結論講反各數則），量紅隊抓回幾個；**改紅隊 batch size 或換模型層級時必重跑**——否則 schema 化／批次化省下的 token 可能是拿偵測率換的而不自知（實測抓 166 錯、但漏了幾個沒有 ground truth 可知，摻毒是唯一能量召回率的方法）。

> **紅隊省不得（收尾重申）**：以上全部只作用在「重複稅、輸出格式、免費零 token 門」，**不用於跳過任何硬門，也不降低敏感科的紅隊抽樣**。答案配對全驗一律全跑、詳解紅隊必跑，這兩道必跑門不因 token 工程而鬆動。

### 2.3.1 模型分層三級判準與不降清單

七槓桿把「省重複稅」對齊「升品質」；再把「哪個環節該用哪級模型」收成一條可判的準則，防兩種誤配：把最強模型燒在腳本能做的事、或把便宜模型放進沒有兜底驗證網的環節。

**三級判準**：

- **腳本能證明的不花 token**——確定性映射、集合比對、規則查表一律零 token 腳本先行（例：送分題＝答案標記為送分符號；官方曾釋疑＝清單查表；選項圈號塌陷＝造字碼位對照表如 `U+E18C–E190＝①②③④⑤` 的確定性映射，先於任何 OCR／看圖——**但對照表是 per-font 的**：那組碼位是特定字型的發現，只適用同源字型的卷，跨卷套用前先驗字型出處）。
- **輸出可被零 token 腳本完整驗證的環節才可下放便宜模型**——格式修、樣式歸因這類「錯了結構 lint／內容守恆檢查抓得到」的活；下放前必須指出兜底驗證網是哪道門，指不出就不准降。**機器處理過≠免抽**：當「驗證」與「轉換」共用同一套啟發式（如選項編號轉字母後用 lint 驗字母覆蓋），那道驗證是警告不是證明——經自動轉換的題照常留在紅隊抽樣池，不因機器處理過而豁免。
- **需要模型「知道對不對」（憑空寫事實）的環節不降**——生成事實、裁決真偽，一律維持中階以上。

**Python 先行原則**：能被前兩層零 token 手段接手的，一律先寫成腳本，剩下的才交模型——這與本節開頭「三層分工」同源（結構 lint 抓格式、檢索錨定防事實錯、紅隊抓剩下的事實錯）。連錯誤類別統計、誤報率、常見雷頻次都走腳本 tally，強模型只留「把統計出的雷歸納成可回灌 prompt 的判斷」那半邊。

**不降清單（審過鎖死，防未來誤降）**：紅隊修改的執行（連帶改動風險無便宜機器可驗）、圖表看圖複判（只開放小樣本 A/B、不直接降）、紅隊本體、新寫線的生成主幹、裁決者、金樣撰寫、申論解析、以及「錯的詳解最傷人」的高風險科（如臨床醫學）——凡「模型必須知道對不對」的環節一律鎖死，不因 token 工程下放。便宜模型只在小樣本 A/B 實測勝出、且該環節有零 token 兜底網時才下放。

## 3. 詳解紅隊（必跑）：反駁式 ＋ 全跑或抽查

> **這道門必跑、不可跳過。** AI 生成**選擇題詳解與申論題解析**之後，一律紅隊查核才能進 build。
> **執行前先問使用者**：要 **全跑**（逐題反駁式查核）還是 **抽查**（下面 §3.2 的統計停止規則）？
> agent 不得自作主張跳過、也不得未問就默認其一。題量小／答案爭議多時建議全跑；題量大且求成本效率時可抽查。
> 申論／短答／中譯英只要是 **AI 生成的解析**就一併納入；官方分級樣卷非 AI 生成物，不在此列。

### 3.1 反駁式（adversarial），不是覆述式

紅隊用最強模型（Opus），任務是**主動找碴、試圖反駁**詳解，而不是把詳解再講一次。具體會挑：

- 正解判定是否寫反（例如把「(D) 正確」寫成「(D) 錯誤」）。
- 措辭是否誤導、把次要當主要。
- 是否引用了過時或錯誤的事實／課綱內容。
- 把握度標太高（`high`）但實際論述站不住。

反駁式比覆述式更能逼出真錯 —— 覆述容易「附和」生成模型的錯誤。

### 3.2 抽查模式的統計停止規則（Wilson CI 動態停）

使用者選**抽查**時（題量大、求成本效率），不逐題用 Opus 重判，改**抽樣**查核並用統計規則決定要不要升級。
（選**全跑**時略過本節，直接逐題反駁式查核。）

> **抽樣錯誤達 k ≥ 4／40 才升級**：每科持續抽樣查核，當累計**真錯**達到 `k ≥ 4`、樣本約 `n = 40`（錯誤率下界 ≥ 4%，以 Wilson 信賴區間下界估計）時，**升級到更強模型重做該科**；否則**續用現模型**繼續生成。

要點：

- 計的是**真錯**（判定寫反、事實錯），不是措辭潤飾這種「為改而改」。
- 用 **Wilson CI 下界**而非單純點估計：小樣本下 Wilson 區間比常態近似穩，下界 ≥ 4% 才算「錯誤率確實偏高」，避免被一兩個離群點誤導升級。
- 是**動態停止**：邊抽邊看，達標就停手升級、沒達標就續抽續生成。不是「先全做完再一次驗」。
- 升級的單位是**該科**，不是整批 —— 某科達標只重做那科，其他科照舊。

### 3.3 抽驗紀錄怎麼存

紅隊發現的真錯，整理成修補檔放在 `data/_stage3/`：

- `fix_reason.json` —— 修紅隊抓到的詳解（以 qid 為鍵，值為新的 `{t, c}`）。
- `fix_empty.json` —— 補空殼（先前漏生成或為空的 qid）。

再由 `scripts/merge_stage3.py` **冪等併回正本**（唯一寫入 `explanations.json` 者），整個 entry 取代、只留 `{t, c}`。

### 3.4 申論／無官方答案題的紅隊角度（錨點不同）

§3.1–3.3 是**選擇題**的紅隊：有官方標準答案當錨，查的是「解析有沒有正確支持那個答案」。但**申論／計算題（如國考會計師、律師申論）沒有官方標準答案**——考選部只公布測驗題標準答案，申論由閱卷委員依評分規準評分、不公布答案。**那個錨點不存在，紅隊角度要整個換掉。**

> ⚠️ **答案配對全驗對申論不適用**：沒有官方答案可逐題配對，那道必跑門只管選擇題。申論的把關全靠下面這套紅隊角度。

**錨點轉移**：從「對齊官方答案」→「**對齊現行權威來源 ＋ 題目要求本身 ＋ 獨立重算**」。
反方人格：**「我是嚴格閱卷委員，要找出這份 AI 參考答案會害考生被扣分或學到錯東西的地方。」**

六個查核角度（依計算型申論的風險排序）：

| # | 角度 | 查什麼 | 錨在哪 |
| --- | --- | --- | --- |
| 1 | **計算重算** | 折舊／分攤／稅額／試算平衡等，**獨立重新算一遍**，不看 AI 給的數字。 | 自行重算（硬對錯） |
| 2 | **準則／法條適用** | 用對準則沒（IFRS 15、IAS 16…）？法條、稅率、函釋對不對？有無套錯科目。 | 對**現行**準則公報／稅法／法規查證 |
| 3 | **題意覆蓋** | 多小題（E1/E2/E3）＋具體要求（「請計算」「依 X 準則說明」）**每項都答到了嗎**？有無答非所問、漏小題。 | 對齊題目 `prompt` 逐項 |
| 4 | **時效失準** | 舊年題的準則／法規已修訂；解析有沒有用**舊版**卻沒標明「以現行為準」。 | 舊年題 × 現行版本差異 |
| 5 | **捏造引用** | AI 編出不存在的準則號／函釋字號／法條／判例。 | 引用逐一驗真 |
| 6 | **把握度誠實** | 爭議題或需現行查證卻標 `high`。 | `c` 級與不確定性是否相符 |

要點：

- **角度 1／2／3 是不同視角**（重算／準則／題意），比派 N 個一樣的紅隊更能抓失效模式（perspective-diverse）；計算題務必**獨立重算**，不複述 AI 的算式。
- **查證來源**：現行準則／法規查證可走權威網站或可用的法規 MCP（如金管會法規、稅法、公司法）；舊年題重點是標清「作答以現行為準」。
- **全跑 vs 抽查仍由使用者定**（§3 開頭）：計算型申論「錯就是硬錯」且舊年版本風險高，建議**角度 1／2 全跑、角度 3–6 可抽查**；但最終由使用者拍板。
- 修補一樣走 §4 節制門 + `_stage3/fix_*.json` + `merge_stage3` 冪等併回（申論併回 `essay_samples.json`／`ref`，紀律相同）。

## 4. 節制門（克制）：只修確認的錯，不為改而改

紅隊抓到問題後，**不是看到什麼都改**。過度修訂會引入新錯、也浪費成本。節制門規則：

1. **只修確認的錯** —— 判定寫反、事實錯、把握度明顯標錯，這些才改。
2. **不為改而改** —— 純措辭偏好、可接受的同義改寫，不動。
3. **改完重 lint、重驗** —— 修補不是終點；改過的 qid 要重跑 lint（標點檢查）並納入下一輪抽驗範圍，確認沒引入新問題。

`merge_stage3.py` 落實了這套克制：它只覆寫修補檔列出的 qid，其餘原封不動；併入後對「改過的 qid ＋ 固定目標範圍」重跑全形標點正規化，再寫回。**最小變動面**是刻意的設計。

## 5. lint：半形／全形標點檢查

### 5.1 規則

中文段落**不可混入半形標點**。中文用全形 `，。、？！「」『』（）：；……——`；英文／數字／日期／URL／程式碼／`§`／縮寫（GSAT／CAP／PWA／MIT／JSON／KV／PDF／HTML／API）一律半形。

lint 是 CI 的一道門（沿用 social-worker 的 `scripts/lint.py` 模板 ＋ `lint.yml`），把「中文句子裡冒出半形 `,` `.` `;` `:` `?` `!`」抓成錯誤。

### 5.2 確定性的全形修正（CJK 後綴規則）

`merge_stage3.py` 內建確定性的全形轉換 `fullwidth()`（純標準庫、無 LLM），規則是 **CJK 字元後緊接的半形子句標點才轉**：

- 主規則正則：`([一-鿿])([,;:?!]|\.(?!\.))` —— 中文字後面接 `, ; : ? !` 或單一 `.`（`\.(?!\.)` 排除省略號 `...`）才轉成全形。
- 補規則：`([)）])(...)` —— 中文子句裡「右括號（半形 `)` 或全形 `）`）＋ 半形子句標點」也算中文語境，補轉（例：`(C誤);社福…` → `(C誤)；社福…`）。

為什麼這條規則安全（**天然受保護**，不會誤轉）：

- **比值** `3:1` —— `:` 前一字元是數字 `3`，非 CJK 也非右括號 → 不轉，保住數學比值語意。
- **數字後標點**、**英文句**、**代號** —— 其標點前一字元為數字或英文字母，非 CJK → 不轉。

這呼應了「全形半形要看狀況」的教訓：理科中英混排是語意問題（誤轉比值冒號、漏轉數字後標點都是錯），所以用 **CJK 前綴**這個保守條件，只動真正的中文語境，**寧可漏轉、不可誤轉**。

### 5.3 順帶正規化

merge 時一併做兩件正規化：

- `c` 值 `medium` → `med`（統一把握度字面值）。
- `meta.source` 過時字串 → 統一新字串；`meta.count` 同步成實際筆數。

## 6. 詳解批次切分（會考為例）

把一科的題目切成可餵給生成模型的批次，由 `scripts/make_cap_expl_batches.py` 處理（會考國文／社會）：

- 來源 `data/_stage/cap_{科}.json`，**只取通過查驗的科目**。
- **文字題**（`needs_figure == False`）每 **20** 題一檔：`capq_text_{i}.json`。
- **圖題**（`needs_figure == True`）每 **12** 題一檔：`capq_fig_{i}.json`（圖題批次小一點，因為單題上下文重）。
- 每檔是題物件陣列，欄位對齊既有 `q_text_*`／`q_fig_*` 格式：
  `qid, subject, year, no, stem, options, answer, passage, needs_figure, figure`。
- **冪等**：每次重建會先 `clear_old` 清掉舊 `capq_text_*`／`capq_fig_*` 再重切；排序鍵為 `(subject, year, no)`。
- **只印計數**：報告印各科文字題／圖題數與批次數，**不印題目內容、不印 base64**。

> 圖架構鐵則（與全專案一致）：圖存獨立 PNG（`data/figures/{檔名}.png`）→ `bank.json` 只存 `figure` 檔名（純文字、可 lint）→ `build_app.py` 建置時才把 PNG 轉 base64 內嵌進 `window.__FIGS__`（`figures_block()` 組成 `{檔名: dataURI}`）。**base64 全程只在建置腳本內部與輸出 HTML 流動，絕不進對話／stdout／Read**。批次切分只搬 `figure` 檔名，碰不到 base64。

## 7. 作文政策：不 AI 代寫，改抓官方分級樣卷

> 拍板決策（見 `docs/DECISIONS.md`）：升學分支**不生成 AI 完整作文**。

### 7.1 不 AI 代寫的範圍

**中文國寫、英文作文**的完整範本，**改抓官方提供的分級樣卷**（從最高分到低分都有），不由 AI 生成完整作文。

理由：

- 作文好壞**高度依賴評分情境**（評分規準、閱卷尺度）。官方分級樣卷比 AI 代寫更可信、更有教學價值。
- 符合「真實資料優先」：讓使用者看官方「6 級分到低級分」的真實範文，比看 AI 編的「標準作文」學得到東西。
- clean-room 一致性：作文範本屬官方機關依其授權提供的素材，不是 AI 生成物。

→ 衍生資料任務：抓 ceec（學測國寫／英文作文評分樣卷）＋ 心測中心（會考寫作 6 級分範文）的官方分級樣卷。

### 7.2 可 AI 整理的範圍

英文的**短答／中譯英**類，**仍可由 AI 整理參考答案**（並一律標明「須查證」）。

理由：短答／翻譯**近乎有標準答案**，不像申論作文那樣依賴評分情境，AI 整理合適。整理出的參考答案沿用第 1 節的標記紀律（標「AI 整理需查證」）。

### 7.3 一張表記住

| 題型 | 處理方式 | 原因 |
| --- | --- | --- |
| 中文國寫（長文） | 官方分級樣卷（最高分→低分） | 依賴評分情境，官方範文更可信 |
| 英文作文（長文） | 官方分級樣卷 | 同上 |
| 英文短答 | AI 整理參考答案（標須查證） | 近乎標準答案 |
| 中譯英 | AI 整理參考答案（標須查證） | 近乎標準答案 |

```
下載(小模型) → markitdown 轉檔(小模型) → parser 忠實解析(逐科 parser)
  → 官方標準答案 find_tables 對應 →【答案配對全驗・必跑：逐題對官方答案，全跑到 0 mismatch】
  → 切批次(make_cap_expl_batches，只印計數)
  → clean-room 詳解生成(選擇題＋申論題；Sonnet；難題/題組/圖式 Opus；標 c 把握度)
  → lint(半形/全形標點，CI 門)
  →【詳解紅隊・必跑：先問使用者「全跑 or 抽查」；抽查走 Wilson 下界 k≥4/40 動態停】
  → 節制門(只修確認的錯，不為改而改) → 修補存 _stage3/fix_*.json
```

## 9. 長跑運維坑（跨模型大批量紅隊的實戰教訓）

巨科（上千則詳解＝數百批）紅隊是多日背景工程，跨越額度重置與多次重啟。以下坑真實踩過，寫進來讓任何人／任何 AI 接手時不重蹈：

### 9.1 「假完成」：批次跑完 ≠ 真的全做完

外部模型（如 Codex/OpenAI）額度**瞬斷**時，紅隊批會「成功啟動但沒產出 verdict」，逐批迴圈把它們記 ERROR 後，仍走到結尾**寫下「ALL DONE」並退出**——但其實缺了幾十批。若下一階段（Opus 修補）信這個 ALL DONE，就會修到不完整的層。

兩道防線，缺一不可：
1. **完整度閘**：判定某層完成**只認 `verdict_b*.json 數 == in_b*.json 數`**，永遠不信 log 裡的「ALL DONE」字樣（它可能是額度斷時寫的假標記）。
2. **有界重試**：跑批的函式跑完一輪後，若仍有缺批就**再跑一輪，最多 N 輪（如 4），且某輪無進展即停**（批本身壞或額度持續斷，別空轉）；唯有真正補滿才寫 ALL DONE。`done` 集合要**排除 ERROR 行**（成功批才算完成，ERROR 批下輪重試）。

### 9.2 resume 要冪等、checkpoint 要逐批

逐批寫 checkpoint（每批一行 `b007 n=25 fix=2`），重跑時讀 checkpoint 跳過已完成批。如此「電腦睡眠／session 變動／額度斷／手動重啟」中斷都無妨，重跑自動接續、不重做、不漏做。**背景進程用 detached（`nohup`）＋ 防閒置睡眠**，否則 session 一變動就被收掉。

### 9.3 額度保險：偵測 → 停 → 等重置 → 自動接

外部模型 5 小時滾動額度燒完時：①**停掉 pool**（別讓它空轉狂記 ERROR）②**清掉卡死的 in-flight 進程**（額度斷時這些呼叫會 hang 成數小時殭屍，務必殺掉，否則佔資源又不產出）③記下重置時間、到點先**測一個極小呼叫**確認通了再重啟。並行隊數（concurrency）越高燒越快——撞額度就降隊數。

### 9.4 shell 斷詞坑（zsh）

手動補跑缺批時，`zsh` **不對未加引號的變數做斷詞**（與 bash 不同）。`NNS="108 109 110"; cmd $NNS` 會把整串當**單一參數**傳入，導致「找不到 in_b108 109 110.json」這種錯。補跑批號要用 `cmd ${=NNS}`（zsh 強制斷詞）、或逐一列出、或用陣列。這個坑會讓「補跑」看似執行其實零產出。
